## 2. 제안 — 제품/운영 결정 필요 (구현 보류)

### 2-1. UX 제안 (우선순위순)

**P1. 카운트다운 압박 완화 (B6)** — 난이도: 하
현재 녹음 시작 직후부터 빨간 굵은 글씨 "남은 시간 N초" + 펄스 점이 상시 노출된다. 만 6-7세에게
시간 압박 신호가 될 수 있다. 제안: 잔여 10초 전까지는 녹음 버튼의 진행 링만 남기고 텍스트를
숨기고, 10초 이하에서만 중립색으로 "곧 끝나요"를 표시. **검사 프로토콜(표준 시행 지침)이 시간
고지를 요구하는지 확인 필요** — 요구한다면 현행 유지.

**P2. 녹음 0회 문항 통과 정책 (B8)** — 난이도: 하
녹음 문항은 한 번도 녹음하지 않아도 [다음]이 활성이다(검토·모달에서 뒤늦게 경고). 아동 오터치
한 번으로 문항이 넘어갈 수 있다. 제안 A: 최소 1회 녹음 전 [다음] 비활성(검사 유연성 감소).
제안 B: 0회일 때 버튼 라벨을 "건너뛰기"로 바꿔 의도적 통과만 허용. **검사 시행 규칙의 문제** —
"응답 거부도 유효한 관찰"이라면 현행 유지가 맞다.

**P3. 관리자 채점 인체공학 (D2+D3)** — 난이도: 중
수십 명 × 26개 녹음 채점 시 클릭 비용이 크다. 제안: ① 이전/다음 아동 키보드 단축키
(예: `[` / `]`), ② "연속 재생" 토글 — 한 녹음이 끝나면 다음 행 자동 재생(AudioBus에 등록 순서
리스트 유지 필요). **채점 프로토콜이 문항별 정지를 요구할 수 있어** 기본 off 토글 권장.

**P4. 마이크 진단 안내 분기 (C2)** — 난이도: 하
권한 거부 안내가 한 줄인데 iOS Safari(설정 앱)와 Android Chrome(주소창 자물쇠)의 해제 절차가
다르다. UA 기반 2-3줄 분기 또는 "도움말 보기" 토글 제안. 문구 정책 확정 필요.

**P5. 검토 복귀 흐름 (E2)** — 난이도: 하
검토에서 문항으로 돌아가 수정한 뒤 [다음]을 누르면 검토로 복귀하지 않고 다음 문항으로 간다.
`from=review`일 때 주 버튼을 "검토로 돌아가기"로 바꾸는 안. 동선 취향 문제라 보류.

**P6. 소소한 것들** — 각 난이도: 하
화면 꺼짐 방지(Wake Lock, 배터리 트레이드오프 · B12), Select 목록 위쪽 플립(뷰포트 하단
잘림 · D9), 문항 이동 시 URL ?q= 동기화(B10), 학교 목록 로딩을 전체 dim 대신 인라인
스피너로(B13), 정렬 가능 헤더 hover 표시(D6), 파형 로딩 스피너(D8), 제출 모달 버튼 라벨
구체화("네/아니오" → "제출하기/돌아가기" — README 수동 체크리스트와 함께 갱신 필요 · D11).

### 2-2. 보안·운영 제안

**S1. 글로벌 로그인 잠금 정책 (F-02 잔여)** — 결정 필요
006 마이그레이션으로 "오답 1회로 잠금 무한 연장"은 막았지만, 공격자가 10분당 50회 실패를
지속하면 신규 관리자 로그인은 여전히 봉쇄된다(기존 8h 쿠키는 유효). 선택지:
(a) 현행 수용 — 단일 관리자 + 세션 8h 전제면 실위험 낮음, (b) 글로벌 버킷을 잠금 대신
점증 지연(백오프)으로 전환 — 가용성↑ 방어력 약간↓. **운영 형태(관리자 수·검사 기간) 기준 판단.**

**S2. 아동 PII 보존 정책·고지 (F-04 잔여)** — 법무 검토 필요
수단(세션 삭제 버튼, 일괄 파기 SQL)은 갖춰졌으나 **정책이 없다**: ① 보존 기한(예: 결과 통보 후
N개월) 확정 → README의 일괄 파기 쿼리를 pg_cron으로 자동화, ② 시작 화면 고지를 수집
항목·목적·보유 기간 명시로 보강(현재 한 줄), ③ 만 14세 미만 법정대리인 동의 절차. 모두
**운영 주체의 법무 판단이 선행**돼야 한다.

**S3. login_attempts 행 정리 자동화 (F-11)** — 난이도: 하
IP도 개인정보성 데이터다. `delete from login_attempts where updated_at < now() - interval '30 days'`를
pg_cron으로 월 1회 실행 권장(README에 수동 절차는 문서화돼 있음).

**S4. nonce 기반 CSP (F-13)** — 난이도: 중
현재 CSP는 frame-ancestors만 통제한다. XSS 벡터가 될 입력 렌더링이 모두 React 텍스트
바인딩이라 실위험은 낮지만, 서드파티 스크립트를 추가하는 순간 방어선이 없다. Next 공식
가이드의 nonce 방식 도입을 **서드파티 스크립트 도입 시점에** 함께 진행 권장.

**S5. 다중 관리자·감사 로그 (N-04)** — 난이도: 상
현재 관리자 credential이 단일 비밀번호라 "누가" 삭제했는지 특정 불가(삭제 시각·대상은 로그에
남김). 연구가 다기관으로 확장되면 관리자 계정 분리 + 행위 감사 테이블 도입 검토.

**S6. 세션 5,000건 초과 대비** — 난이도: 중
목록 API가 5,000행 상한(도달 시 경고 로그)이다. 대상 규모가 수천 명을 넘길 계획이면 서버
페이지네이션 + 검색·필터의 서버 이관이 필요하다. 현 규모(수백)에서는 불필요.

**S7. 인프라 이전 시 IP 신뢰 재검토** — 메모
`x-real-ip` 신뢰는 Vercel 전제다. 다른 인프라로 옮기면 프록시가 이 헤더를 덮어쓰도록
구성해야 한다(`lib/request.ts` 주석에 명시).

> **CSV 내보내기**: 도입하지 않기로 결정(2026-07-17). 외부 집계 요구가 없어 제외한다.
